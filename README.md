# KeibaLab

競馬予想サイト（単一HTML＋Node製プロキシ＋Supabase）。

## 起動

```
npm install
npm start        # http://localhost:3001 （node proxy.js）
```

`proxy.js` が `index.html` の配信と netkeiba 系APIの中継を行う。
JRA-VAN（JV-Link）からの過去データ取り込みは [tools/README.md](tools/README.md) を参照。

## 総合予想（旧「馬柱・血統・騎手」の3軸を統合）

予想軸の切り替えUIは廃止し、「総合予想」1本に統合した。
**血統・騎手の情報を削除したのではなく、総合予想を構成する特徴量として扱う。**

- 参考スコア（0-100点）は従来互換のまま表示する（内部的には旧・馬柱軸の重みで算出。
  保存済みの検証データ・キャッシュとの互換のため）
- スコアの構成要素は `computeScoreComponents(h)` に分離した純粋関数で算出する:
  `form`(近走) / `career`(通算) / `fit`(馬場・距離適性) / `venueFit`(同競馬場) /
  `agari`(上がり) / `jockey`(騎手) / `kinryo`(斤量) / `sireFit`(血統・産駒) / `market`(市場)
  。欠損は null（0=低評価と区別）

## 単勝・期待値判定

予想ページで買い方に「単勝」を選ぶと、**期待値にもとづく購入／見送り判定**を表示する。

- 推定勝率・適正オッズ・期待回収率・購入／見送り判定を全頭ぶん表示
- 期待回収率が閾値未満のレースは「このレースは見送り」と明示する（買わない判断が最重要機能）
- 推奨は原則1レース1点（購入候補のうち期待回収率が最大の1頭のみ）

### 推定勝率：総合勝率モデル（市場オフセット付きsoftmax）

```
z_f,i      = (特徴量f_i - レース内平均) / レース内標準偏差   … 標準偏差0はz=0
             欠損は z=0 ＋ missing_f=1 のフラグ（「欠損＝平均」と「本当に平均」を区別）
logit_i    = log(marketProb_i) + Σ_f β_f・z_f,i + Σ_f γ_f・missing_f,i
p_i        = softmax(logit)_i                               … 数値安定化・レース内合計1
```

- **係数が全て0なら市場確率と厳密に一致する**（市場がベースライン）
- 係数は正負どちらも許容（市場が過大評価する特徴は負になる）
- 使用特徴量: 従来7特徴量に、距離変化・馬場替わり・昇降級・休養／叩き・枠／脚質／コース適性を加えた18特徴量、
  さらに坂路調教4特徴量（本数・最速時計・最終追い切りからの日数・終い加速）を加えた22特徴量＋各欠損フラグ
  - `market` はオフセット側で使用済みのため特徴量にしない
  - `sireFit` は学習データ（JV-Data）側で取得できず**学習時と実運用時で条件が異なるため不使用**（保存はする）
- 実装: `index.html`「総合勝率モデル」区画（純粋関数）。Node側は `lib/win-model.js` が抽出して共有

### 学習

```
npm run train:win    # tools/train-win-model.js → models/win-model.json
```

- 損失: 勝ち馬に対するレース単位の交差エントロピー ＋ L2正則化
- 最適化: 全バッチ勾配降下（初期値0＝市場確率から開始。決定的・乱数不使用）
- ハイパーパラメータは固定（lr=0.5 / epochs=500 / L2=1e-4）。**組み合わせ探索はしない**
- **学習期間: 2015-01-01〜2021-12-31 のみ。2022年以降のデータでは係数を一切学習しない**
- 学習後、調整確認期間（2022-2023）で市場単独と比較し、下記の採用条件を自動判定して
  `models/win-model.json` に係数・特徴量・学習期間・生成日時・バージョン・採用可否を保存する

### モデル採用条件（調整確認 2022-2023 で判定）

1. Log Loss が市場単独より改善
2. Brier Score が市場単独より悪化していない
3. 予測勝率帯と実勝率が大きく逆転していない
4. 期待値が高い帯ほど極端に実回収率が下がる現象がない
5. NaN・確率合計の不整合がない

条件を満たさない場合はUIへ採用せず、**市場確率ベースへフォールバック**して
「市場確率ベースの暫定推定」と表示する（失敗結果もこのREADMEに記録する）。
モデルの配信は `proxy.js` の `action=win-model`（固定パスのみ返す。任意ファイル配信はしない）。

### 期待値判定（固定・変更しない）

```
fairOdds            = 1 / predictedWinProb
effectiveOdds       = 現在オッズ × ODDS_SAFETY_FACTOR(0.95)
expectedRoiPercent  = predictedWinProb × effectiveOdds × 100
```

- `expectedRoiPercent >= VALUE_BET_THRESHOLD(120)` → 購入候補、それ以外は見送り
- オッズが取得できない馬 → 判定不能
- 購入候補が複数なら期待回収率最大 → 予測勝率 → 馬番小の優先で1頭だけ（1レース最大1点）
- 購入候補が0頭なら**レースごと見送り**

## バックテスト（時系列分割・3モデル比較）

```
node tools/jv-import.js --dir jvdata2000 --min-date 2015-01-01   # 特徴量つきデータ生成（数分）
npm run train:win                                                 # モデル学習＋採用判定
npm run backtest:win                                              # 3モデル比較
```

期間は次で固定:

- **学習・設計**: 2015-01-01〜2021-12-31（係数の学習に使用）
- **調整確認**: 2022-01-01〜2023-12-31（採用判定のみに使用。学習には不使用）
- **過去評価**: 2024-01-01〜データ最終日
  - ※ 2024年以降は既存方式の結果確認ですでに参照済みのため、**完全未使用テストではない**。
    「過去評価」と呼ぶ。真の前向きテストは今後蓄積するレースで行う

比較対象: ①市場確率のみ ②旧スコア補正方式（logit+0.20×スコアz） ③総合モデル。
指標: Log Loss（レースごとの勝ち馬の -log p の平均）/
**Brier Score（全馬の (予測勝率 − 勝敗0or1)² の平均。小さいほど良い）** /
1位予測的中率 / 予測勝率帯の校正 / 期待回収率帯の実回収率 / 購入件数・的中率・回収率。
**回収率だけでモデルを選ばない。**

### 2026-07-15 時点の結果

学習されたモデル（win-model-v1）は従来7特徴量に、条件変化11特徴量を追加した。
係数の符号はレース内標準化後の市場オッズに対する補正方向であり、単独の因果効果とは解釈しない。

| 期間 | LogLoss 市場→総合 | Brier 市場→総合 | 1位的中 市場→総合 |
|---|---|---|---|
| 調整確認 2022-2023 | 1.91064 → **1.90806** | 0.057898 → **0.057827** | 33.6% → **34.2%** |
| 過去評価 2024〜 | 1.91078 → **1.90930** | 0.057412 → **0.057364** | 33.6% → **33.6%** |

- **採用判定: ✅ 条件変化統合モデルを採用**（現行モデルと市場の両方を調整確認期間で上回った）
- 旧スコア補正方式は全期間で市場単独より悪い（LogLoss/Brierとも）→ UIから除外済み
- 期待回収率120%以上は調整確認8件（0的中）、2024年以降11件（1的中・回収率141.8%）。
  母数が極端に少ないため、回収率改善の証明とは扱わない
- 参考: 全レースで◎の単勝を固定購入した場合の回収率は 72.5〜73.5%（＝買うほど損）

### 走破タイム・着差・対戦クラスの追加検証（2026-07-15）

JV-DataのSEから、従来読み捨てていた走破タイム・1着馬とのタイム差・コーナー順位を抽出する。
過去走だけから次の候補特徴量を生成し、現行採用モデルと同じ期間・固定ハイパーパラメータで比較した。

- `speedForm`: 同日結果を使わず、前日までの同コース条件の走破タイムに対して標準化した直近5走の速度指数
- `marginForm`: タイム差を距離1000m当たりへ換算した直近5走の着差パフォーマンス
- `classLevel`: 直近5走の対戦クラス（未勝利〜G1）

|---|---:|---:|---:|---|
| 現行採用モデル | **1.90942** | **0.057852** | **1.91012** | 維持 |
| ＋着差 | 1.90950 | 0.057854 | ― | 不採用 |
| ＋対戦クラス | 1.90966 | 0.057859 | ― | 不採用 |
| ＋速度指数 | 1.90971 | 0.057854 | 1.91026 | 不採用 |
| ＋速度・着差・クラス | 1.90982 | 0.057858 | ― | 不採用 |

候補はいずれも市場単独（1.91064）より良かったが、現行採用モデルより悪化したため本番モデルには入れていない。

`lib/condition-features.js` の `summarizeWeight`）。当日体重はレース約50分前の発表値で予想時点で利用可能。

| モデル | 2022〜2023 LogLoss | Brier | 判定 |
|---|---:|---:|---|

### 坂路調教特徴量の追加検証 → 本番採用（2026-07-19）

JV-Linkの坂路調教データ（dataspec=SLOP・HCレコード、2014年〜約598万本）を新規に取得し
（`tools/jv-fetch.ps1 -DataSpec SLOP -Types HC`、出力は `jvtrain/HC.txt`）、
`lib/training-features.js` で4特徴量を実装した。すべてレース当日より前の調教のみ使用する。

- `trainCount28`: レース前28日間の坂路調教本数（÷6・上限2）
- `trainBestSpeed`: 同期間の最速4Fタイムを、トレセン別ローリング分布（前日まで・直近3000本）で標準化したz
- `trainLastGap`: 最終追い切りからの日数（÷14・上限2）
- `trainAccel`: 最速調教の終い加速ラップ（lap2−lap1秒・±1.5にクランプ）

| モデル | 2022〜2023 LogLoss | Brier | 1位的中 | 2024〜 LogLoss | Brier | 判定 |
|---|---:|---:|---:|---:|---:|---|
| 市場単独 | 1.91064 | 0.057898 | 33.6% | 1.91078 | 0.057412 | 比較基準 |
| 旧採用モデル（①〜⑤統合） | 1.90806 | 0.057826 | 34.2% | 1.90930 | 0.057364 | 置換 |
| ＋坂路調教4特徴量 | **1.90582** | **0.057771** | **34.3%** | **1.90884** | **0.057359** | **本番採用** |

これまでで最大の単独改善幅。係数を見ると**時計の速さ自体（trainBestSpeed β≈0）は市場に織り込み済み**だが、
調教本数（β=+0.025）・終い加速（β=+0.037）・追い切り間隔（β=−0.016）に市場が拾っていない情報があった。
購入成績は調整確認24件（回収率40.4%）・2024年以降19件（82.1%）と依然母数不足で、回収率の証明にはならない。

**実運用の注意（学習時と運用時の条件合わせ）:**

- `proxy.js` が `jvtrain/HC.txt` の直近120日分を読み、netkeibaの馬ID（=JRA血統登録番号）で結合して
  ライブの出馬表にも同じ特徴量を付与する
- HCデータはレース当日朝の差分取得で最新化する:
  `jv-fetch.ps1 -From <前回取得日時> -OutDir jvtrain -DataSpec SLOP -Types HC -Append`
- データが古い／無い場合は調教特徴量が全馬nullになる。**全馬一様の欠損フラグはsoftmaxで打ち消し合う**ため
  予測は壊れず、調教なしモデル相当へ自然に縮退する（Supabase版は調教特徴量を供給しないため常にこの縮退挙動）
- ウッド調教（dataspec=WOOD・WCレコード、75.5万本）も取得済みだが特徴量は未実装（次の実験候補）

### 制約（重要）

- バックテスト・学習の単勝オッズは**確定オッズ**。発売中に同じオッズで買える保証はない
- 推定勝率は市場オッズを補正した統計的推定であり、**勝率や回収率を保証するものではない**
- 閾値120%・安全率0.95・学習期間は固定。**過去評価期間（2024〜）を見て係数・閾値・特徴量を調整してはいけない**
- モデルの真価は今後蓄積するレースでの**前向きテスト**でのみ確認できる
